# Valuation Audit Reference

本 Skill 为**参考知识库**，不包含自动化执行脚本。

Agent 在面对以下场景时应自动参考对应的 checklist：

**Excel 底稿审核** → `references/valuation_audit_checklist.md`：
- 用户让 Agent 审核 DCF 模型 / Excel 底稿
- 用户让 Agent 检查估值分析的合理性
- 用户让 Agent 做可比公司/案例分析
- 用户让 Agent 审查 EBITDA 正常化调整
- 用户让 Agent 做尽职调查 / Due Diligence 清单
- 用户让 Agent 检查三表模型（IS/BS/CF）联动关系

**Word 报告审核** → `references/word_report_audit_checklist.md`：
- 用户让 Agent 审核评估报告（Word）
- 用户让 Agent 审核评估说明（Word）
- 用户让 Agent 检查报告与底稿的一致性
- 用户让 Agent 核对企业工商信息（iFinD 场景）
- 用户提及"报告审核"、"Word审核"、"报告模板"、"特别事项"、"评估依据"等关键词

**通用触发**：用户提及"评估"、"估值"、"底稿审计"、"尽调"等关键词时，根据上下文判断使用哪个 checklist

**Agent 输出必须遵循三级格式**：🔴 Fatal → 🟡 Sanity Check → 🔵 Best Practice

**Agent 必须先做可见性筛选（Step 0）**：大型底稿常有 50%+ 隐藏Sheet/行/列，仅审核可见区域。详见 checklist §0.2。

**Agent 必须遵循三阶段审核流程**（§0.2）：
1. **阶段一（结论验证）**：从结论表入手，验证最终结论的逻辑链
2. **阶段二（逐表深度审核）**：一级明细表 → 二级辅助表 → 会计处理理解，逐行核对公式、引用、符号规约
3. **阶段三（来源充分性评估）**：假设依据审计 → 覆盖完整性 → 管理层偏向测试

## NotebookLM 知识库查询规范

### 定位

NotebookLM 知识库（"工作"笔记本）是本 Skill 的**专家会诊通道**，不是搜索引擎。
知识库内容需用户自行准备，根据自身专业需求选择覆盖范围（如会计准则、评估法规、行业基准等）。

### 知识分区原则（绝对红线）

| 知识来源 | 用途 | 禁止用途 |
|:--|:---|:---|
| Agent 内部知识 | 审计逻辑推理、发现可疑点、公式校验 | ❌ 引用具体条文编号、提供行业基准数据 |
| checklist (references/) | 审计要点、常见错误清单、操作规范 | — |
| NotebookLM 知识库 | 准则条文原文、行业基准、方法论细节 | ❌ 回答 checklist 已覆盖的基础问题 |

**核心规则：Agent 内部知识只用于推理，条文/数据/方法论原文必须来自 checklist 或 KB。两者混用 = 幻觉风险源。**

### 触发条件（仅以下情况才查询）

1. **checklist 明确标注"不覆盖"的领域**：成本法/基础法、中国评估准则条文细节
2. **需要具体准则条文编号支撑审计发现**：如"中评协专家指引第X号第Y条"、"证监会评估类第X号"
3. **特殊会计处理需要准则原文**：CAS 21/IFRS 16、金融工具、合并报表等
4. **行业特定估值方法/基准**：checklist §2.3 行业表未覆盖的领域
5. **监管红线的具体法规依据不确定**：checklist §8 只有方向性指引时

**严禁查询**：checklist 已明确覆盖且无争议的问题（如WACC公式、DCF步骤、EBITDA定义）。但如果某个看似"基础"的概念在特定场景下存在交互效应（如"折旧分摊"在"毛利率倒推成本"模型中的估值影响），则属于**合格的专家级问题**——关键是问题中必须包含完整的场景上下文。

### 成功查询范例

以下为实战中验证有效的查询，供 Agent 参考问题精度和场景上下文的写法：

| 查询问题 | 触发场景 | KB 返回要点 |
|:--|:--|:--|
| "收益法预测中其他应收款其他应付款如何区分经营性和非经营性" | 审核WC时发现其他应收占收入33%但预测期维持不变 | 模拟抽离法、关联方往来原则上非经营性、因企而异原则 |
| "折旧摊销分摊至成本和期间费用的区别及对企业价值评估的影响" | 发现成本用毛利率倒推，折旧分摊位置可能影响EBIT | 付现/非付现成本分离原则、毛利率倒推下折旧"隐身"风险 |
| "在建工程对收益法预测期资本性支出的影响及审核要点" | 资产负债表有CIP但CapEx预测仅含更新支出 | CIP后续投入必须全额纳入、产能-收入匹配、借款费用资本化停止条件 |
| "WACC中无风险利率的选取标准和条文依据" | 需要条文编号支撑审计发现 | 中评协〔2020〕38号第8-9条、证监会评估类第1号、10年期国债YTM |

### 底层依赖

- **notebooklm-py >= 0.4.0** — [github.com/teng-lin/notebooklm-py](https://github.com/teng-lin/notebooklm-py)
- v0.4.0 新增：`--json` 结构化输出（含 source_id 引用）、`--profile` 多账号支持
- 安装/升级：`pip install --upgrade notebooklm-py==0.4.0`

### 调用方式

```
python ~/.agents/skills/valuation-audit-reference/scripts/query_knowledge_base.py "查询问题"
python ~/.agents/skills/valuation-audit-reference/scripts/query_knowledge_base.py "查询问题" --json    # 结构化输出
python ~/.agents/skills/valuation-audit-reference/scripts/query_knowledge_base.py "查询问题" --profile myprofile
```

- 响应时间：约 80-90 秒/次，偶尔因网络超时返回 `Chat request timed out`
- 超时处理：**原样重试同一问题**，不要为了缩短而删减关键上下文词。精度比速度更重要。超时大概率是网络/服务器波动，不是问题长度问题
- 查询应**异步执行**（run_in_terminal async），Agent 继续审核其他模块，稍后取回结果
- 每次查询应为**精准的专家级问题**，而非基础概念检索。宁可一次问透，不要反复追问同一主题

### 嵌入四阶段审核流程

| 阶段 | 查询时机 | 查询类型示例 |
|:--|:---|:---|
| **阶段一（结论验证）** | 识别到 checklist 不覆盖的评估方法/特殊资产类型时 | "碳资产/数据资产评估的中评协最新指引要求" |
| **阶段二（逐表审核）** | 遇到特殊会计处理，checklist §1.4/§8 不够具体时 | "CAS 21使用权资产在DCF中的折旧与租赁付款还原处理" |
| **阶段三（来源充分性）** | 需要条文支撑审计发现，或行业基准验证假设时 | "XX行业收入倍数合理区间的权威数据来源" |
| **阶段四（输出前）** | 发现项引用了条文编号但无来源 → 补充查询 | "证监会评估类第1号关于资产组认定的具体条文" |

### 输出处理

- KB 返回的答案中带 `[数字]` 引用编号 → 在审计报告发现项中标注为 `[KB: 来源名称]`
- 发现项引用格式：`[Checklist §X.Y]` 或 `[KB: 中评协专家指引第12号]`
- **严禁** `[Agent 已知]` 或无出处引用

### 输出前自检门（Stage 4 之前强制执行）

1. 统计：多少个发现项引用了 `[Checklist]`？多少引用了 `[KB]`？
2. 如果有发现项引用了**具体条文编号**（如"CAS 21第X条"、"中评协〔2020〕38号"）但无 `[KB]` 来源 → Agent 必须回退查询验证
3. 涉及 checklist "不覆盖"领域的发现项，如果无 `[KB]` 来源 → 标记为 `🔵 需人工确认：超出知识库覆盖范围`

### 审核完成后的 Checklist 迭代建议

审计报告输出完成后，Agent 可以向用户**建议**将本次审核中发现的、checklist 尚未覆盖但具有通用性的审计逻辑固化到 checklist。但：
- **决定权在用户**：Agent 仅提出建议，由用户判断该发现是通用规律还是偶发特殊
- **建议格式**：列出发现项、建议写入的 checklist 章节、建议条目内容
- **严禁 Agent 自行修改 checklist**：所有 checklist 修改必须经用户确认后执行
- 如需准则条文打磨建议条目措辞，可使用 KB 查询获取权威表述
